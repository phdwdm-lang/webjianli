## 文档用途

这份文档用于在真正修改代码前，先把“旧页快照层切换”这条路线讲清楚，帮助确认三件事：

1. 这条方案是否真的能解决当前的核心问题。
2. 实现时需要改动哪些模块，是否会影响现有页面结构。
3. 验收时应该重点看什么，避免再次出现“目标页 -> 目标页”的错误观感。

这是一份**实施前方案文档**，当前阶段**不执行代码修改**。

---

## 1. 当前问题回顾

目前导航切换已经从浏览器原生 `View Transition API` 调整为内容区自管的上下滑动，但仍然存在一个核心问题：

- 用户点击导航后，目标页会过早以“真实页面”的形式出现在屏幕上。
- 因为目标页已经提前活跃起来，用户主观上看到的是“目标页已经来了，然后目标页继续动”。
- 最终切换观感会退化成“目标页 -> 目标页”，而不是“当前页 -> 目标页”。

换句话说，当前问题不是动画方向错了，而是：

**目标页出现得太早，旧页没有被稳定地冻结为一张完整画面。**

---

## 2. 核心目标

本方案希望把切换体验收敛到更接近 PPT 翻页的感觉：

1. 点击导航时，先保留一份“旧页静态画面”。
2. 旧页静态画面按方向滑出。
3. 目标页在下方或上方准备好，再滑入。
4. 导航栏全程固定，不参与页面画面的快照替换。
5. 不出现“目标页先出现，再继续过渡”的观感。

---

## 3. 方案核心思路

### 3.1 不再依赖“旧页 React 树”做 outgoing 面板

当前自管双面板的问题在于：

- outgoing 不是旧页快照，而是旧页 React 节点
- incoming 不是静态承接层，而是立即活跃的新页面

本方案要改成：

- outgoing 使用“旧页快照层”
- incoming 使用“真实目标页”

这样页面切换时，用户看到的上半场主要是：

- 一张冻结的旧页画面滑走

而不是：

- 一个还在继续参与 React 生命周期的旧页组件

---

### 3.2 快照层的本质

这里的“快照层”不是重新启用浏览器原生 `View Transition API`，而是在我们自己的内容容器内部创建一层静态视觉副本。

推荐做法：

- 在触发导航切换前，获取当前内容容器的 DOM 节点
- 克隆该节点的可见结构，或将其复制为一个独立的展示层
- 这层快照只负责显示，不参与交互、不参与 React 更新
- 导航切换时，让这层快照执行“滑出”动画

这意味着：

- 旧页在视觉上被冻结
- 新页可以在后台完成路由切换和挂载
- 用户不会先感知到目标页在动

---

## 4. 为什么这条方案更适合当前问题

### 4.1 它直接针对“目标页出现过早”

当前最难受的问题不是导航本身，而是目标页的第一帧时机。

只要旧页不是一个稳定的视觉快照，而目标页又是真实页面，那么就容易出现：

- 先看到目标页
- 再看到目标页继续做切换

快照层方案的优势在于，它直接把旧页固定成一张“不会再变化的画面”，因此能从根上削弱这个问题。

### 4.2 它比再次尝试浏览器原生 View Transition 更可控

之前浏览器原生方案出现的问题是：

- 导航栏会被顶层快照机制影响
- 导航可能闪烁、消失、重新出现

而快照层方案把快照范围限制在**内容区内部**，导航仍然保持真实 DOM，不参与快照层。

这正符合当前需求：

- 内容切换有强过渡感
- 导航全程常驻

### 4.3 它比“继续压新页内部动画”更稳

另一条备选路线是：

- 保留现在的双面板机制
- 再逐页关闭新页内部动画

但这个方案的问题是：

- 工作会比较碎
- 每个页面都可能有例外
- 未来新增页面时还可能复发

快照层方案虽然底层一点，但逻辑更统一，长期维护会更稳定。

---

## 5. 推荐实现结构

### 5.1 保留当前的大结构

以下结构不建议推翻：

- `AppChrome`
- `RouteTransitionProvider`
- `RouteTransitionShell`
- `SideNav`
- `MobileNav`

原因是当前导航已经被抽离到外层，这是对的。

本次只需要把 `RouteTransitionShell` 从“旧页 React 双面板”升级为“旧页快照层 + 新页真实层”。

---

### 5.2 RouteTransitionShell 的新职责

调整后，`RouteTransitionShell` 需要承担以下职责：

1. 提供一个明确的内容区根容器 `contentStageRef`
2. 在导航切换前，基于当前内容区创建旧页快照层
3. 路由跳转后，将真实新页渲染到内容区
4. 同时驱动：
   - 旧页快照层滑出
   - 新页真实层滑入
5. 动画结束后销毁快照层，恢复正常状态

---

### 5.3 状态设计建议

建议在 `RouteTransitionProvider / RouteTransitionShell` 中明确区分以下状态：

- `idle`
  - 没有正在切换
- `capturing`
  - 正在生成旧页快照层
- `navigating`
  - 路由已触发，新页正在进入
- `animating`
  - 旧页快照层和新页层正在做位移动画
- `error`
  - 快照捕获失败，降级为无动画直接切换

当前实现里只有 `idle / animating`，对这条路线来说粒度偏粗。

增加这些中间状态的意义是：

- 方便分清楚“什么时候创建快照”
- 方便分清楚“什么时候允许新页显示”
- 方便避免过早收尾

---

## 6. 快照层实现建议

### 6.1 推荐方案：DOM 克隆层

建议优先采用 DOM 克隆层，而不是 canvas 截图或图片渲染。

做法：

1. 取到内容区根节点
2. `cloneNode(true)` 克隆当前可见内容结构
3. 将克隆节点挂到一个专门的 overlay 容器中
4. overlay 容器绝对定位覆盖在内容区上方
5. overlay 容器禁止交互，仅用于显示旧页

推荐理由：

- 实现路径相对直接
- 不依赖额外截图工具
- 更容易和现有 CSS 结构对齐

禁止交互的实现方式：

```css
.snapshot-overlay {
  pointer-events: none;
  user-select: none;
}
```

### 6.2 需要注意的边界

DOM 克隆层不是 React 节点，因此要明确：

- 不能让克隆层参与任何状态更新
- 不能保留原有交互事件
- 不能让克隆层影响布局流

它应当只是一个短时存在的视觉层。

**补充注意事项：**

1. **滚动位置**：`cloneNode` 不会复制元素的 `scrollTop` / `scrollLeft`，如果内容区有滚动，需要在克隆后手动同步滚动偏移。
2. **CSS 动画**：克隆节点可能重新触发 CSS `@keyframes` 动画，建议在克隆后给 overlay 添加 `animation: none !important` 或移除相关 class。
3. **Canvas / Video**：这些元素内容不会被 `cloneNode` 复制，若页面包含此类元素需额外处理或接受视觉差异。
4. **无障碍与重复 id**：克隆层会复制原有 `id`、`aria-*` 关系和可聚焦节点，overlay 应设置 `aria-hidden="true"`，必要时增加 `inert`，避免读屏和焦点系统把快照层当成真实页面。

## 7. 新页显示策略

### 7.1 新页不应在“裸状态”下直接暴露

推荐做法：

- 路由完成后，新页真实内容进入内容容器
- 但切换中显示层级应严格控制：
  - 旧页快照层在上
  - 新页真实层在下
- 新页真实层只执行整体的滑入，不要先裸露地被用户看到

同时需要补充一个实现约束：

- 路由切换时应显式接管滚动行为，例如优先使用 `router.push(targetPath, { scroll: false })`
- 主转场结束后，再决定是否将窗口滚动重置到目标页顶部

如果不接管这一步，浏览器可能在路由切换时先把窗口滚到顶部，导致快照层和真实内容层出现错位。

### 7.2 新页内部动画的处理建议

即使采用快照层方案，仍建议新增一个简单约束：

- 在模块切换过程中，新页内部首屏入场动画应尽量静默
- 等主转场结束后，再恢复正常页面内动效

这样可以进一步减少：

- 新页卡片自己先动
- 和主切页动画抢视觉焦点

这个约束不必作为第一阶段强制实现，但应写进方案中作为预留优化点。

---

## 8. 导航栏的职责边界

本方案下，导航栏的职责非常明确：

- 始终保持真实 DOM
- 不参与旧页快照生成
- 不参与内容区滑动层
- 点击后立即高亮目标模块
- 转场中允许 hover / 高亮反馈
- 继续限制重复路由触发

也就是说：

**导航只负责控制，不负责成为被动画处理的对象。**

## 9. 实施步骤建议

### 阶段一：结构改造

目标：

- 将当前“旧页 React 双面板”切换改为“旧页快照层 + 新页真实层”

主要工作：

- 在 `RouteTransitionShell` 中增加内容区根引用
- 新增快照层容器
- 增加快照层生命周期管理

### 阶段二：时序改造

目标：

- 保证旧页快照先创建，再触发路由切换

主要工作：

- 在导航点击时先进入 `capturing`
- 快照层准备完成后再执行 `router.push`
- 路由完成后进入 `animating`

**超时保护建议：**

- 不建议把 `cloneNode(true)` 的兜底写成“100ms 超时中断”，因为 DOM 克隆本身是同步操作，定时器无法中途打断
- 更合理的做法是：
  - 在快照前先做轻量前置判断，例如容器缺失、页面处于异常状态、存在已知不可克隆内容
  - 若前置判断失败，直接进入 `error` 状态，降级为无动画切换
  - 对实际克隆耗时做埋点记录，用于后续优化，而不是寄希望于运行中超时中断
- 路由切换时同步接管窗口滚动，避免快照层先于页面滚动发生错位

### 阶段三：动画改造

目标：

- 让旧页快照层滑出、新页真实层滑入

主要工作：

- 定义内容区 overlay 层动画
- 调整层级关系
- 统一 forward / backward 方向规则

### 阶段四：细节优化

目标：

- 压低卡顿感，减少与页面内动效冲突

主要工作：

- 统一过渡时长
- 必要时静默新页内部首屏动画
- 处理 tooltip / hover 的切换节奏

## 10. 验收标准

### 10.1 核心观感

- 切换时用户首先感知到的是“当前页在滑走”
- 不再出现“目标页先出现，再继续切换”的观感
- 不再出现“目标页 -> 目标页”的感觉

### 10.2 导航表现

- 导航在切换过程中不消失
- 导航不闪烁
- 导航不重新挂载
- 导航高亮在点击后立即切换到目标模块

### 10.3 过渡流畅度

- forward / backward 方向与模块顺序一致
- 内容区无明显空白停顿
- 切换过程中无明显掉帧或突兀重绘
- 路由切换期间窗口滚动位置不应提前跳变

### 10.4 稳定性

- 快照层能在动画结束后正确销毁
- 不会残留旧页 overlay
- 不会阻塞后续导航切换

## 11. 风险与注意事项

### 11.1 风险一：DOM 克隆可能复制过大的内容层

如果内容区过大、图片很多，克隆层可能带来额外性能负担。

缓解思路：

- 快照层只覆盖主内容容器，不要克隆整个 `body`
- 控制快照层生命周期，动画结束立即销毁

### 11.2 风险二：克隆节点的样式依赖上下文

如果某些页面样式强依赖父级上下文，克隆层可能出现显示偏差。

缓解思路：

- 快照层挂载在内容区同级或近邻位置
- 尽量继承当前容器的尺寸和布局上下文

### 11.3 风险三：页面内复杂交互元素在快照层中不可用

这其实是预期行为，但需要明确：

- 快照层只负责“看起来像旧页”
- 不需要也不应该继续交互

### 11.4 风险四：滚动位置丢失

`cloneNode(true)` 不会复制元素的滚动状态。

缓解思路：

- 克隆后遍历需要同步滚动的容器，手动设置 `scrollTop` / `scrollLeft`
- 或者只对内容区根容器做滚动同步（通常足够）

---

## 12. 结论

这条方案的价值在于：

- 它不是继续微调现有双面板时序
- 而是直接把“旧页没有被冻结”这个根因拿掉

如果目标是尽量稳定地消除“目标页 -> 目标页”的错误观感，这条路线比继续调整现有动画时机更值得优先尝试。
